## 폴더 목적 및 구조

이 폴더는 QBT(Quant BackTest) 프로젝트의 **개발/운영 문서**를 모아두는 곳입니다.

### 폴더 구조

```
docs/
├── CLAUDE.md           # docs 관련 규칙의 소스 오브 트루스 (이 문서)
├── plans/              # 중간 이상 스코프 변경을 위한 단계별 구현 계획서
│   ├── _template.md    # 계획서 템플릿
│   └── PLAN_*.md       # 실제 계획서
└── archive/            # 완료/폐기된 계획서 및 작업 문서
```

### 원칙

- 문서는 "문서 자체"가 목적이 아니라, **안전한 변경(테스트/품질 게이트/롤백)**을 돕기 위한 도구입니다.
- 작은 변경(오타/주석/사소한 버그)은 계획 문서 없이 바로 처리합니다.
- 큰 변경(여러 모듈에 영향, 데이터 스키마/저장 규칙 변경 등)은 `plans/`에 계획을 먼저 작성합니다.

## AI 작업 흐름

이 섹션은 AI를 이용해 QBT를 수정/확장할 때, **일관된 품질**과 **안전한 변경**을 보장하기 위한 최소 규칙을 정의합니다.

> **가장 중요한 규칙**:
> **어떤 폴더/패키지 작업이든 해당 폴더의 `CLAUDE.md`를 먼저 읽고 따릅니다.**
> (루트 `CLAUDE.md` + 도메인별 `CLAUDE.md`)

### 계획서가 필요한 경우

아래 중 하나라도 해당하면 `docs/plans/`에 계획서를 먼저 작성합니다.

- 변경 범위가 **여러 파일/여러 모듈**에 걸친다
- `storage/*` CSV 스키마, 저장 규칙, 결과 컬럼 등 **불변 규칙**에 영향이 있다
- 백테스트/시뮬레이션 핵심 로직(수익률, 비용, 체결 규칙 등)을 바꾼다
- 테스트 추가가 다수 필요하거나, 회귀 가능성이 높다
- "어디까지가 done인지"가 애매하다

작은 변경(단일 버그 픽스/주석/로그/포맷)은 계획서 없이 바로 진행합니다.

### 표준 작업 절차

#### 0) 규칙/맥락 로딩

- 루트 `CLAUDE.md` 확인
- 작업 대상 폴더의 `CLAUDE.md` 확인 (예: `src/qbt/tqqq/CLAUDE.md`)
- 데이터 관련이면 `utils/data_loader.py`, `common_constants.py` 규칙 확인

#### 1) (필요 시) Plan 작성

**계획서 작성 절차**:

1. **먼저 이 문서(`docs/CLAUDE.md`)를 읽습니다**
   - AI 작업 흐름
   - 품질 게이트 커맨드
   - 프로젝트 불변 규칙

2. **`docs/plans/_template.md`를 기반으로 계획서 작성**
   - 파일명: `docs/plans/PLAN_<short_name>.md`
   - Phase는 3~7개로 분해
   - 각 Phase마다 Validation 명시

3. **계획서 내용**:
   - Goal: 목표 설정
   - Non-Goals: 범위 제외 항목
   - Context: 배경/필요성/영향 받는 규칙
   - Definition of Done: 완료 조건 체크리스트
   - Scope: 변경 대상 파일 및 데이터/결과 영향
   - Phases: 단계별 계획 (각 Phase별 Validation)
   - Risks: 예상 위험 및 완화 전략

#### 2) Phase 단위 구현

각 Phase는 다음을 만족해야 다음 Phase로 진행합니다.

- 구현 코드 + 테스트 코드(또는 기존 테스트 보강)
- 포맷/린트/테스트(아래 "품질 게이트") 통과
- Plan 문서의 체크박스 업데이트

#### 3) 마무리

- Plan의 "완료 조건(Definition of Done)" 충족 확인
- 필요한 경우 `README.md`/도메인 `CLAUDE.md`/문서 업데이트
- 회귀 위험이 있는 변경이면 테스트 케이스를 반드시 남깁니다.

**완료/폐기된 계획서 처리**:
- 완료되거나 폐기된 계획서는 `docs/archive/`로 이동
- 히스토리 관리 목적이며, 향후 유사 작업 시 참고 자료로 활용

## 품질 게이트

로컬에서 아래 중 최소 1개 이상을 실행해 확인합니다(변경 규모에 따라 조합).

### 테스트

```bash
# 전체 테스트
./run_tests.sh

# 커버리지 포함
./run_tests.sh cov
```

### 린트/포맷

```bash
# 린트
poetry run ruff check .

# 포맷 검사
poetry run black --check .
```

> **주의**: AI는 사용자 환경에서 테스트를 직접 실행할 수 없으므로,
> 실행 결과는 사용자가 확인하고 피드백(에러 로그)을 공유합니다.

## plans 폴더 사용 규칙

이 폴더는 "중간 이상 스코프 변경"을 위한 **단계별 구현 계획서**를 저장합니다.

### 파일 네이밍

- 템플릿: `docs/plans/_template.md`
- 계획서: `docs/plans/PLAN_<short_name>.md`
  - 예: `PLAN_tqqq_cost_model_refactor.md`
  - 예: `PLAN_backtest_grid_results_schema.md`

### 언제 쓰나?

아래 조건이면 계획서 작성:

- 여러 모듈/여러 파일에 걸친 변경
- CSV 스키마/저장 규칙 변경
- 백테스트/시뮬레이션 핵심 계산 로직 변경
- 리팩토링(구조/아키텍처) 작업
- 테스트 대량 추가/수정이 필요한 작업

단일 파일의 사소한 변경은 계획서 없이 진행합니다.

### 템플릿 사용법

`docs/plans/_template.md`를 복사하여 새 계획서를 작성합니다.

**템플릿 구성**:
1. **상단 메타정보**: 상태(Draft/In Progress/Done), 작성일, 범위, 관련 문서
2. **1) Goal**: 목표 설정
3. **2) Non-Goals**: 범위 제외 항목
4. **3) Context**: 배경/필요성/영향 받는 규칙
5. **4) Definition of Done**: 완료 조건 체크리스트 (6개 항목)
6. **5) Scope**: 변경 대상 파일 및 데이터/결과 영향
7. **6) Phases**: 단계별 계획 (3~7개 Phase, 각 Phase별 Validation)
8. **7) Risks**: 예상 위험 및 완화 전략
9. **8) Notes**: 참고 사항

### Phase 구성

Phase는 **3~7개**로 구성하며, 각 Phase는:
- 명확한 목표와 범위를 가짐
- 독립적으로 검증 가능
- Validation 섹션에 검증 커맨드 명시

### 운영 방식

계획서는 "문서 작성"이 목적이 아니라:
- **리스크 분해(Phase)**
- **검증 커맨드 명시**
- **롤백(되돌리기) 전략**

를 명확히 해서 변경을 안전하게 만드는 것이 목적입니다.

## archive 정책

`docs/archive/` 폴더는 다음을 보관합니다:

- **완료된 계획서**: 성공적으로 완료된 PLAN_*.md 파일
- **폐기된 계획서**: 취소되거나 중단된 계획서
- **작업 중 생성된 임시 문서**: 테스트 진행 상황, 요약 문서 등

### 목적

- 히스토리 관리
- 향후 유사 작업 시 참고 자료로 활용
- 현재 진행 중인 계획서와 분리

### 규칙

- 완료/폐기된 계획서는 사용자가 수동으로 이동
- 파일명은 그대로 유지 (예: `PLAN_tqqq_cost_model_refactor.md`)
- 필요시 완료 날짜를 파일명에 추가 가능 (예: `PLAN_tqqq_cost_model_refactor_20240101.md`)

## 프로젝트 불변 규칙 요약

변경 작업 시 반드시 지켜야 하는 핵심 규칙입니다. 자세한 내용은 루트 `CLAUDE.md`를 참고하세요.

### 계층 분리

- CLI 계층(`scripts/`)은 **얇게 유지**: 인자 파싱/로깅/예외 처리/도메인 호출만
- 도메인 로직(`src/qbt/`)은 **ERROR 로그 금지**, 예외는 `raise`로 전파

### 데이터 처리

- CSV 로딩은 `utils/data_loader.py`를 통해서만(중앙집중)
- 저장 규칙(`storage/*`)은 루트 `CLAUDE.md` 기준을 우선

### 코딩 표준

- 경로는 **Path 객체만 사용**
- 메시지/문서/주석은 **한글**
- 모든 함수에 타입 힌트 필수
- Google 스타일 Docstring (한글)

### 상수 관리

- 공통 상수: `common_constants.py`
- 도메인 상수: 각 도메인의 `constants.py`
- 명명 규칙: `COL_`, `KEY_`, `DISPLAY_`
- 중복 금지
