# hold_days 고원 분석 — 전체 자산 그리드 서치 실행 계획

> 작성일: 2026-03-08
> 목적: 코딩 AI에 전달하여 hold_days 0~10 그리드 서치를 구현·실행하도록 한다.
> 전제: CLAUDE.md(backtest 도메인)를 숙지한 상태에서 진행한다.

---

## 1. 배경과 목적

### 1.1 왜 이 실험이 필요한가

버퍼존 전략의 hold_days 파라미터(돌파 후 N일 연속 확인 조건)에 대해, 현재까지 hold=0, hold=2, hold=3 세 개 포인트의 성과만 확인되어 있다. QQQ 기준으로:

```
hold=0:  Calmar 0.200
hold=2:  Calmar 0.210
hold=3:  Calmar 0.300
```

3개 포인트만으로는 hold=3이 "고원(plateau)의 일부"인지 "봉우리(spike)"인지 판단할 수 없다. 고원이면 hold=3은 안정적인 선택이고, 봉우리면 과최적화 의심이 된다. 이 형태를 확인하기 위해 0~10 범위의 그리드 서치가 필요하다.

### 1.2 실험의 목적

hold_days = [0, 1, 2, 3, 4, 5, 7, 10]의 8개 값에 대해, 7개 자산(QQQ, SPY, IWM, EFA, EEM, GLD, TLT) 전체에서 Calmar 곡선의 형태를 확인한다.

이 실험은 **"최적값 찾기"가 아니라 "고원 형태 확인"**이다. 결과에서 Calmar 1위를 선택하는 것이 아니라, 고원이 존재하는지·어디에 위치하는지를 판단하는 데 사용한다.

### 1.3 기대 결과물

7개 자산 × 8개 hold_days = 56회 백테스트 실행 후, 아래 형태의 비교 테이블을 생성한다.

```
         hold=0   1      2      3      4      5      7      10
QQQ      0.200   ?      0.210  0.300  ?      ?      ?      ?
SPY      0.350   ?      0.460  ?      ?      ?      ?      ?
IWM      0.060   ?      0.100  ?      ?      ?      ?      ?
EFA      0.210   ?      0.200  ?      ?      ?      ?      ?
EEM      0.090   ?      0.050  ?      ?      ?      ?      ?
GLD      0.200   ?      0.250  ?      ?      ?      ?      ?
TLT      0.020   ?      0.020  ?      ?      ?      ?      ?
```

이 테이블의 ?를 전부 채우는 것이 이 실험의 산출물이다.

## 3. 탐색 범위

### 3.1 hold_days 값

```
hold_days = [0, 1, 2, 3, 4, 5, 7, 10]
```

- 0~5: 1일 단위로 세밀하게 (핵심 구간)
- 7: 1주일. 주간 주기 효과 확인
- 10: 2주. 확인 기간이 너무 길면 추세 초기 수익을 얼마나 잃는지 확인

### 3.2 대상 자산 (7개)

| Ticker | 시그널 소스 | 매매 대상 | 검증 역할 |
|---|---|---|---|
| QQQ | QQQ | QQQ | 기준 자산 |
| SPY | SPY | SPY | 최고 성과 자산, 고원 확인 핵심 |
| IWM | IWM | IWM | 소형주 (가짜 돌파 빈번) |
| EFA | EFA | EFA | 선진국 |
| EEM | EEM | EEM | 신흥국 (hold=2에서 유일 악화) |
| GLD | GLD | GLD | 비주식 자산 |
| TLT | TLT | TLT | 채권 |

### 3.3 총 실행 횟수

7개 자산 × 8개 hold_days = **56회** 백테스트

## 4. 제약사항

### 4.1 절대 제약

1. **hold_days 이외의 파라미터를 변경해서는 안 된다.** 이 실험은 hold_days 단일 변수의 효과만 측정하는 통제 실험이다.
2. **결과에서 "최적값"을 선택하는 로직을 넣지 않는다.** 이 실험의 목적은 최적화가 아니라 형태 확인이다.
3. **기존 전략의 결과 파일을 덮어쓰지 않는다.** 이번 실험의 결과는 별도 경로에 저장한다.

### 4.2 결과 저장 원칙

- 56회 각각의 summary.json(또는 동등한 성과 지표)이 보존되어야 한다.
- 최소한 자산명, hold_days, CAGR, MDD, Calmar, 거래수, 승률이 기록되어야 한다.
- 최종 비교 테이블은 터미널에 출력하고, 동시에 CSV 파일로도 저장한다.

## 6. 구현 방향성

### 6.1 핵심 구조

기존 버퍼존 전략 백테스트 엔진을 그대로 사용하되, hold_days 값을 외부에서 주입하여 반복 실행하는 구조로 만든다. 새 전략을 만드는 것이 아니라, 기존 엔진에 파라미터만 바꿔가며 돌리는 래퍼 스크립트이다.

### 6.2 실행 흐름

```
1. 7개 자산 × 8개 hold_days 조합 목록 생성
2. 각 조합에 대해:
   a. 고정 파라미터 + 해당 hold_days로 백테스트 실행
   b. 결과(CAGR, MDD, Calmar, 거래수, 승률) 수집
3. 전체 결과를 비교 테이블로 정리
4. 터미널 출력 + CSV 저장
```

### 6.3 기존 코드 활용

- 이 프로젝트에는 이미 버퍼존 전략의 백테스트 엔진이 구현되어 있다.
- 기존 `BufferZoneConfig` 기반 통합 전략 모듈과 `run_single_backtest.py` 스크립트가 있다.
- CLAUDE.md(backtest 도메인)에 전략 모듈 구조, config 패턴, 파라미터 오버라이드 방식이 설명되어 있으므로 반드시 먼저 읽고 기존 구조를 파악한 뒤 구현한다.

### 6.4 구현 시 주의사항

- 기존 전략 결과 디렉토리(`storage/results/backtest/buffer_zone_*`)를 건드리지 않는다. 이번 실험 결과는 별도 경로(예: `storage/results/backtest/hold_days_plateau/`)에 저장한다.
- 각 자산의 데이터 파일 경로는 `common_constants.py`에 이미 정의되어 있을 수 있으므로 먼저 확인한다.
- 기존 hold=0(3P) 결과와 hold=2(4P) 결과가 이미 저장되어 있더라도, 이번 실험에서 동일 조건으로 새로 실행하여 통일된 조건의 결과를 생성한다.

## 7. 실행 후 확인 사항

### 7.1 검증 포인트

실행 결과에서 아래 기존 데이터와 일치하는지 확인한다 (동일 조건 재실행이므로 같은 값이 나와야 한다).

| 확인 항목 | 기대값 |
|---|---|
| QQQ hold=0 Calmar | 0.200 |
| QQQ hold=2 Calmar | 0.210 |
| QQQ hold=3 Calmar | 0.300 |
| SPY hold=0 Calmar | 0.350 |
| SPY hold=2 Calmar | 0.460 |
| GLD hold=0 Calmar | 0.200 |
| GLD hold=2 Calmar | 0.250 |

위 값들과 크게 다르면 파라미터 설정이나 실행 조건에 오류가 있는 것이므로, 결과를 신뢰하기 전에 원인을 파악해야 한다. 소수점 둘째자리 수준의 미미한 차이는 라운딩에 의한 것이므로 허용한다.

### 7.2 결과 보고

실행이 완료되면 §5에서 정의한 형식(비교 테이블 + CSV)으로 결과를 정리하여 보고한다. 결과에 대한 해석이나 판단은 포함하지 않아도 된다 (해석은 별도로 진행한다).
