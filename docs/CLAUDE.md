# docs 폴더 가이드

> 이 문서는 `docs/` 폴더의 문서 작성 및 **계획서(Implementation Plan) 운영 규칙**을 정의합니다.  
> 프로젝트 전반의 공통 규칙은 [루트 CLAUDE.md](../CLAUDE.md)를 참고하세요.

## 폴더 목적

`docs/`는 QBT 프로젝트의 개발/운영 문서와 변경 계획서(Implementation Plan)를 관리합니다.

## 폴더 구조

```
docs/
├── CLAUDE.md           # docs 관련 규칙 (이 문서)
├── plans/              # 변경 계획서 저장소
│   ├── _template.md    # 계획서 템플릿(예시)
└── archive/            # 과거 문서 보관소(기본 무시)
```

### docs/archive 정책 (중요)

- `docs/archive`는 과거 문서 보관용입니다.
- 현재 작업의 규칙/근거/소스 오브 트루스(Source of Truth)로 사용하지 않습니다.
- 현행 규칙은 **루트 `CLAUDE.md` + 작업 범위 도메인 `CLAUDE.md` + 최신 plan**만을 기준으로 합니다.

## 소스 오브 트루스 & 반드시 읽어야 할 문서

계획서/코드를 작성할 때는 아래 문서를 **반드시 먼저 읽고**, 그 안에 기재된 규칙을 **전체 숙지**한 뒤 작업합니다.

- 공통 규칙: `CLAUDE.md`(루트)
- 도메인 규칙: 작업 대상 경로의 `CLAUDE.md`

  - 예: `src/qbt/backtest/CLAUDE.md`, `src/qbt/tqqq/CLAUDE.md`, `src/qbt/utils/CLAUDE.md`, `scripts/CLAUDE.md` 등

- 테스트 규칙: 테스트를 추가/수정한다면 `tests/CLAUDE.md`를 반드시 따릅니다.

> ⚠️ 계획서 안에 “영향받는 규칙”을 작성할 때는
> “규칙을 나열”하는 대신, **참고할 문서(파일) 목록을 나열하고 그 문서에 적힌 규칙을 모두 준수**한다고 명시합니다.

## 포맷/린트/테스트 규칙

### Black 실행 원칙

- `poetry run black .`는 **마지막 Phase에서 실행**합니다.
- 단, `poetry run black --check .`가 실패한다면:

  - “마지막 Phase에서 포맷 적용 예정”으로 기록할 수는 있지만,
  - 최종 Done/완료 선언 전에 반드시 `poetry run black .`를 실행해 포맷을 맞춥니다.

### Ruff 실행 원칙

- 각 Phase Validation에서 `poetry run ruff check .`를 실행합니다.
- 오류가 나오면 **해당 Phase에서 즉시 수정 후 재검증**합니다.

### 테스트 실행 원칙 (스킵 금지)

- 각 Phase Validation에서 `./run_tests.sh`를 실행합니다.
- 실패하면 **해당 Phase에서 즉시 수정 후 재검증**합니다.

#### ✅ 스킵(Skipped) 테스트 절대 금지 (강제 규칙)

- 어떤 이유로든 테스트를 `skip` 상태로 남겨서는 안 됩니다.
- `./run_tests.sh` 결과에 **skipped가 1개라도 있으면**:

  - 그 Phase는 “완료/성공/그린 유지”로 선언할 수 없고,
  - plan 전체도 **✅ Done 처리할 수 없습니다.**

- 해결이 어렵다면:

  - Done 처리를 하지 말고 **상태를 In Progress로 유지**하거나,
  - “왜 어려운지/무엇이 막히는지/어떤 선택지가 있는지”를 기록한 뒤
  - **새 plan으로 분리**하여 해결합니다.

- “스킵해두고 나중에”는 허용하지 않습니다.

## plans 폴더 사용 규칙

### 파일 네이밍

- 계획서는 `docs/plans/PLAN_<short_name>.md` 형태로 생성합니다.
- `<short_name>`은 작업 범위와 목적이 드러나도록 간결히 작성합니다.

  - 예: `PLAN_backtest_core_execution_rules.md`

## 계획서 작성 방법 (AI 모델이 반드시 따를 것)

- 새 계획서는 `docs/plans/_template.md`를 복사해 작성합니다.
- `_template.md`는 “예시 템플릿”이며, 특정 도메인 전용이 아닙니다.
- `_template.md`를 수정하거나 새로운 형태의 계획서를 만들고 싶다면, **반드시 이 문서(`docs/CLAUDE.md`) 규칙을 포인터로 두고 준수**해야 합니다.

### 1) 계획서에 반드시 포함해야 하는 섹션

계획서는 아래 구성을 반드시 포함합니다.

- Goal: 목표 설정
- Non-Goals: 범위 제외 항목
- Context: 배경/필요성/영향 받는 규칙(파일 목록으로 “전체 숙지” 선언)
- Definition of Done: 완료 조건 체크리스트 (아래 7개 항목)

  - 기능 요구사항 충족
  - 회귀/신규 테스트 추가
  - 테스트 통과 (`./run_tests.sh`) **+ skipped=0**
  - 린트 통과 (`poetry run ruff check .`)
  - 포맷 적용 (`poetry run black .`) — 마지막 Phase에서 실행
  - 필요한 문서(README/CLAUDE/plan) 업데이트
  - 계획서 체크박스 최신화(Phase/DoD/Validation 모두 반영)

- Scope: 변경 범위(변경 대상 파일, 데이터/결과 영향)
- Phases: 단계별 계획(각 Phase의 할 일 + Validation)
- Risks: 리스크와 완화책
- Notes: 메모/결정사항/링크

### 2) Phase 통합/분리 규칙 (문맥 기반)

> Phase는 “파일 수”가 아니라 **문맥(context)** 기준으로 구성합니다.

- 한 Phase 안에서 “검증/수정/재검증”이 자연스럽게 닫히는 단위로 묶습니다.
- 정책/인바리언트를 테스트로 고정해야 한다면, **Phase 0(레드)**를 별도로 둡니다.
- Phase 1부터는 **그린(전체 테스트 통과 상태)** 유지가 원칙입니다.

  - 이때 그린은 **pass만 허용**하며, **skipped=0**을 포함합니다.

### 3) 승인(Approval) 규칙

- 간단한 계획(Phase 1 + 마지막 Phase 정도)이라면 중간 승인 없이 진행할 수 있습니다.
- Phase 2 이상으로 늘어나는 경우에는 **Phase 전환 시점마다 승인 요청 블록이 있어야 합니다.**

#### 승인 요청 시 필수 포함(체크박스/증거 게이트)

승인 요청을 올리기 전, 아래를 **반드시** 수행/기록합니다.

- (필수) 해당 Phase의 체크박스를 plan 파일에서 최신 상태로 업데이트([ ]→[x])
- (필수) 이번 Phase에서 [x]로 바뀐 항목 목록을 요약
- (필수) Validation 결과 요약(실행 커맨드 + 통과/실패)
- (필수) 테스트 결과에는 **passed/failed/skipped 수를 반드시 포함**

  - skipped가 존재하면 승인 요청을 “완료 기반”으로 올릴 수 없으며, 스킵 제거 후 다시 요청합니다.

> 체크박스 업데이트 없이 “완료/성공/Done/진행해도 될까요?” 같은 승인 요청을 하면 안 됩니다.

### 4) “완료(Done)” 선언 기준 (서술 금지, 증거 기반)

- `./run_tests.sh` 통과는 **필요조건**일 뿐, 체크리스트/DoD가 미완이면 Done이 아닙니다.
- ✅/Done 표시는 “감상”이 아니라 “증거 기반 완료”를 의미해야 합니다.
- 특히 **skipped가 1개라도 있으면 Done 금지**입니다.

  - 가능한 증거: 실행 커맨드 결과, 변경 파일 목록, 핵심 diff 요약, 커밋 메시지/식별자

### 5) 스코프 드리프트(계획↔실행 불일치) 방지 규칙

작업 중 “리스크/사정”으로 **원래 체크리스트 항목을 수행하지 않기로 결정**했다면:

- 기존 plan을 “완료”로 꾸미지 않습니다.
- 선택지:

  1. 해당 항목을 미완료로 남기고, 이유/후속 계획을 Notes 또는 Risk에 기록(체크리스트 구조 변경 금지)
  2. 구조/규칙/목표 자체를 바꿔야 한다면 **새 plan**을 작성(기존 plan은 체크리스트 업데이트만)

### 6) 이미 만들어진 계획서 수정 금지 (중요)

- 이미 생성된 `docs/plans/PLAN_<short_name>.md`는 **체크리스트 업데이트 외에는 수정하지 않습니다.**
- (예: 진행 상황 체크, Done 처리, 날짜 업데이트 등은 가능)
- 체크리스트 업데이트란: [ ] -> [x], 상태/날짜 갱신, 진행 로그 추가만 포함(서술/규칙/구조 수정은 금지).
- 내용/구조/규칙을 바꾸고 싶으면 **새 계획서로 작성**합니다.

### 7) 커밋 메시지(Commit Messages) 규칙

커밋 메시지는 plan 작성 시점부터 계획서 안에 함께 작성합니다.

- 커밋 메시지 형식 규칙:

  - 반드시 `기능명 / ` 형태로 시작
  - 간결하고 명확한 한국어 표현 사용
  - 변경사항의 핵심 내용 반영
  - 추정(상상)으로 쓰지 말고, 해당 Phase에서 실제로 변경하는 범위를 기준으로 작성

- 파일 경로 기반 기능명 추천 규칙(권장)

  - `src/qbt/backtest/` 변경: `백테스트 / `
  - `src/qbt/tqqq/` 변경: `TQQQ시뮬레이션 / `
  - 그 외는 변경 내용을 보고 그때그때 적절한 기능명을 선택한다.

- 계획이 복잡한 경우(Phase 2 이상):

  - 승인 요청이 있는 모든 Phase 전환 지점마다

    - `승인 요청` 블록 + `Commit Messages (Phase N)`(3개)이 세트로 반드시 존재해야 합니다.

  - 마지막 Phase에는 반드시 `Commit Messages (Final)` 섹션을 두고 종합 커밋 메시지 3개를 추가 작성합니다.

## 완료 후 문서 처리

- 완료/폐기된 계획서는 사용자가 수동으로 `docs/archive`로 이동할 수 있습니다.
- 파일명은 유지합니다.
